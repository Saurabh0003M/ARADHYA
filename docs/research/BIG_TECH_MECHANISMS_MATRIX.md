# Big-Tech Mechanisms Matrix for Implemented Aradhya Features

This file compares implemented Aradhya behavior against official big-tech or official OSS mechanisms that solve the same class of problem.

Status note:

- This matrix started as the pre-optimization baseline for the April 10, 2026
  indexer pass.
- After landing the first P0 runtime changes, refresh any row that discusses
  exact-match lookup, miss amplification, or `.txt` scan overhead before using
  it as the current-state snapshot.

Scope rules:

- Only implemented Aradhya features are included.
- Official sources come first.
- Third-party analysis is intentionally excluded from the matrix body.
- "Current Big-O" is a repo-derived engineering estimate from the current implementation, not a formal proof.

Source type legend:

- `open implementation`: official public repo, wiki, or code shows the mechanism.
- `official API only`: official API or documentation describes the capability, but not the internal engine.
- `closed internal implementation`: official docs describe behavior and architecture, but source code is not public.

Adopt decision legend:

- `adopt now`: low-regret mechanism to add in the next optimization pass.
- `adapt later`: useful, but not the next bottleneck to solve.
- `watch`: keep the concept in mind; current Aradhya design is already aligned or the gap is not urgent.
- `reject`: intentionally do not copy this mechanism for the current repo.

## Immediate Conclusions

- Aradhya's biggest gap is not that it lacks a cache. It already has a cache. The real issue is that cache freshness and miss handling still fall back to expensive rebuild behavior.
- The most defensible P0 borrowings are watcher-assisted invalidation, negative caching, and stronger exact-match indexing. These have direct support from official Windows, VS Code, and Watchman mechanisms.
- The planner boundary is already architecturally sound. The improvement target is stricter schema transport, not replacing the deterministic-first design.
- The voice path is correct but intentionally simple. Throughput features such as bounded concurrency or partial-result streaming are secondary to fixing the indexer hot path.
